# TODO Rules

## 1. Purpose

本文档定义根目录 `TODO.md` 的任务格式、协作用法、删除规则、提交规则和 PR 收口规则。

目标是让 `TODO.md` 只承载未关闭任务队列，不承载方案正文、执行记录、长篇论证或未经审阅的执行许可。

## 2. Bounded Context

`TODO.md` 是未关闭任务面板。

TODO、commit 和 PR 固定组成一套协作模型：

- TODO 表达当前还要做什么。
- Commit 表达一次具体工程判断或小步能力变化。
- PR 表达阶段性交付边界，必须完整、可验证，并完成文档、TODO 和 RUNBOOK 收口。

Commit 可以是中间态，但必须有明确判断含义，不得是随机保存点。

## 3. TODO Types

`TODO.md` 承载三类任务：

- 拆解任务：目标已经确定，需要按治理文档或 RUNBOOK 拆成执行任务。
- 执行任务：已经达成共识，可以进入实现、测试、文档同步和提交。
- 待讨论项：需要人类先做决策，不能直接执行。

拆解任务和待讨论项不是执行许可。未经人工确认的拆解任务和执行任务固定放入 `待审阅任务项`，AI 不得执行 `待审阅任务项`。

## 4. File Structure

`TODO.md` 固定使用以下结构：

```markdown
# TODO List

## 说明

- `TODO.md` 是未关闭任务面板，不是完成历史。
- 宏观任务必须先讨论边界，再拆解为可执行 TODO。
- 已完成任务必须删除，不在 `TODO.md` 中打勾保留。
- 完成历史保留在 commit 或 PR 中。

## 当前任务项

- [ ] `范围对象`：任务标题
    - 任务类型：执行任务
    - 依据文档：`docs/.../XXX.md`
    - 范围对象：精确文件路径、模块或对象
    - 处理动作：一句话说明要做什么
    - 验收点：可检查的完成结果
    - 重要度：N/10

## 待审阅任务项

- [ ] `范围对象`：任务标题
    - 任务类型：拆解任务或执行任务
    - 依据文档：`docs/.../XXX.md`
    - 范围对象：精确文件路径、模块或对象
    - 处理动作：一句话说明要做什么
    - 验收点：可检查的完成结果
    - 重要度：N/10

## 待讨论项

- [ ] 需要决策的问题
    - 任务类型：待讨论项
    - 关联任务：任务名或范围对象
    - 决策要求：需要人工确认的具体问题
    - 重要度：N/10
```

没有任务时，对应章节可以为空，但不得记录已完成任务清单。

## 5. Task Item Rules

任务固定使用 `- [ ]`。

执行任务必须包含：

- `任务类型`
- `依据文档`
- `范围对象`
- `处理动作`
- `验收点`
- `重要度`

规则：

- `范围对象` 必须能稳定定位模块、文件或治理域。
- `依据文档` 必须指向稳定规则、RUNBOOK 或需求来源。
- 一个执行任务只表达一个主动作。
- `处理动作` 只写一个明确动作。
- `验收点` 只写可检查结果。
- `重要度` 使用 `N/10`。
- 单项任务文件过多时必须继续拆分。

不得在任务项中写完整方案、长篇背景、预设提交信息、长命令教程、已完成任务清单或无关扩展步骤。

## 6. Collaboration Rules

写入 TODO 前，固定判断：

1. 该任务是否值得进入 `TODO.md`。
2. 该任务是否能用一句话说明问题或目标。
3. 该任务是否能归类为拆解任务、执行任务或待讨论项。
4. 该任务是否不是低置信度观察项。
5. 该任务是否存在明确的执行动作、拆解依据或决策问题。

拆解执行任务前，固定判断：

1. 拆解任务或待讨论项是否已经经过人机讨论。
2. 拆解后的 TODO 是否需要人工审阅。
3. 每个执行 TODO 是否能独立验收。
4. 每个执行 TODO 是否能对应到代码、测试、文档或提交动作。
5. 执行顺序是否合理。

## 7. Deletion And Closure Rules

完成任务后固定处理：

- 完成整项：删除该 TODO。
- 完成部分：收窄为剩余未完成内容。
- 发现拆分错误：重写为正确的待审阅任务。
- 待讨论项完成决策：删除待讨论项；如仍需执行，新增明确执行任务或拆解任务。

删除或收窄 TODO 必须发生在真正完成该 TODO 的 commit 中。如果 commit 只是中间判断，不删除 TODO。

收口前固定判断：

1. 实现目标是否已经达到可检查状态。
2. 当前改动是否仍存在未完成范围。
3. 当前改动是否影响测试、文档或任务入口。
4. 当前工作区中是否混入其他不相关修改。
5. 当前 commit 是否能用一句话解释“这次记录了什么判断”。

## 8. Test Check Rules

收口前必须检查当前改动影响了哪些测试层次。

最小检查清单：

- 是否新增或改变公开入口、协议签名、校验或异常响应。
- 是否改变运行时编排、状态流转、跨模块调用或结果模型。
- 是否改变规则、配置、脚本或外部集成行为。
- 是否改动架构规则、命名规则或其他可能影响静态门禁的内容。

行为变化已发生时，必须同步补充或更新对应测试。只有纯文档、机械改名、注释或格式整理，且不影响行为和规则时，可以不新增测试。

## 9. Document Sync Rules

收口前必须检查当前改动是否要求同步文档。

最小检查清单：

- 治理规则或固定做法变化：检查 `docs/00-governance/*.md`。
- 文档结构变化：检查 `docs/AGENTS.md` 和 `docs/00-governance/DOCUMENT-RULES.md`。
- PR 验证入口变化：检查 `docs/00-governance/PR-RULES.md`。
- 组件盘点或库能力变化：检查对应 inventory、readiness 或设计文档。

改动影响文档定义的固定口径时，必须同步文档。仅实现补齐、但不改变既有文档口径时，不顺手改无关文档。

## 10. Commit Rules

commit 粒度固定围绕工程判断组织。一个 commit 表达一次具体判断、小步能力变化、测试锁定、配置装配或文档规则落点。

commit 可以是阶段任务的中间态，但必须满足：

- 判断含义明确。
- 改动范围可解释。
- 不混入无关修改。
- 不故意留下不清楚的破坏。
- 涉及可运行入口时，运行最小相关验证。

commit message 固定格式：

```text
Type(scope): 中文说明
```

`Type` 固定使用以下白名单：

- `Feat`：新增用户可见能力。
- `Fix`：修复缺陷。
- `Docs`：文档、治理、说明。
- `Style`：纯格式或样式调整，不改变行为。
- `Refactor`：重构，不改变行为。
- `Test`：测试新增或调整。
- `Build`：依赖、构建、脚手架、CI。
- `Chore`：维护性变更，非源码、非交付能力。

`scope` 固定使用以下白名单：

- `app`：React 应用入口、页面、运行时 UI。
- `styles`：Tailwind、全局样式、样式配置。
- `build`：Vite、TypeScript、package、pnpm、lint/test 工具链。
- `governance`：`AGENTS.md`、`docs/00-governance/`、PR/TODO 规则。
- `docs`：普通文档、盘点、README。
- `test`：测试支撑和测试用例。

新增 `Type` 或 `scope` 必须先更新本文档，再用于 commit message。

一个 commit 只能使用一个 `Type(scope)`，不得用多个 type 或多个 scope 共用一个提交。需要同时写 `Docs/Build`、`app,styles`、`build+test` 等组合时，必须拆成多个 commit，或选择唯一能表达本次工程判断的主 `Type(scope)`。

检查清单：

- `Type` 是否来自白名单。
- `scope` 是否来自白名单。
- 是否只有一个 `Type(scope)`。
- 中文说明是否点出具体能力，避免只写“调整”“修改”“优化”。

## 11. PR Rules

PR 固定表示阶段性交付边界。

PR 必须满足：

- 阶段任务完整。
- 可验证。
- 对应验证命令已运行或明确说明不适用。
- 行为、接口或流程变化已同步文档。
- 相关 TODO 已清空或收窄为剩余未完成内容。
- 已完成且无剩余价值的 RUNBOOK 已清理。
- PR 描述记录验证命令和结果。

PR 描述入口由 `.github/pull_request_template.md` 承载；PR 规则由 `PR-RULES.md` 承载。
